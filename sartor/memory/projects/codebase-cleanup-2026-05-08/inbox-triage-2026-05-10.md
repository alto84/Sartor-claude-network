# Inbox triage — 195 flagged → 0

PLAN.md Phase F follow-up. After the `dedup_status: already_landed` filter fix on 2026-05-10, the curator's drain count dropped from "would pollute 25 targets" to clean. But the same run was still flagging 195 "unparseable / missing required fields" entries every pass, generating noise and burning curator runtime.

Triage diagnosed: the inbox was conflating two unrelated uses.

## What the inbox actually contains

**Use #1 — Curator input.** Proposed-memory entries with strict schema (`id`, `origin`, `created`, `target`, `operation`). Auto-generated by the per-machine extractors (e.g., `rocinante-extractor` in `personal-data-gather`). Live in `inbox/{peer}/proposed-memories/{YYYY-MM-DD}/`. These are what the curator is for.

**Use #2 — Peer-to-peer memos.** Free-form markdown with whatever frontmatter the author needed: `type: inbox-message`, `type: peer-loop-report`, `type: phone-home`, `type: drift-detector-output`, `type: wifi-health-report`, `type: report`, etc. Used for status notes, mission docs, peer alerts, cron-task output. These were landing at `inbox/{peer}/` root level, which the curator walks. Curator can't parse them as curator input → flags them → never advances.

195 flagged entries was 195 conflated peer-memos that the curator was re-flagging every run.

## What changed structurally

### 1. Curator code — RESERVED_DIRS expanded

`sartor/curator_pass.py` line 46-58: added `_memos`, `_curator_logs`, `_specs`, `loop-reports` to the reserved-dir set. The curator's `_iter_entry_files` walker now skip-walks files under any of these.

`_memos` is the new canonical "not curator input" subdir per peer.

### 2. Bulk-moved 168 entries into `_memos/`

153 moved by the bulk script + 15 partial-schema/unparseable individually moved. Final destinations:

- `inbox/rocinante/_memos/wifi-health/` — 122 wifi-health-report .md files (15-min audit-trail snapshots from the WiFi monitor)
- `inbox/rocinante/_memos/registry-drift/` — 8 registry-drift-detector outputs (every-4h snapshots from check-registry.py)
- `inbox/rocinante/_memos/` — 8 misc Rocinante peer memos + 6 partial-schema items (phone-homes, sartor-agent-os peer memo, vastai-management review)
- `inbox/gpuserver1/_memos/` — 15 gpuserver1 memos including MISSION docs, peer-self-loop-prompt, gateway-cron-decision report, heartbeat, rtxserver-vastai-onboarding procedure, and the gpuserver1 `_heartbeat.md`
- `inbox/rtxpro6000server/_memos/` — 4 memos including MISSION-finetune-loyalty, PHONE-HOME-phase-1-sanity-failure, abliteration-onboarding peer-directive, peer-self-loop-prompt

168 files all moved cleanly. Move-not-delete per archive-not-collapse — git tracks each as a rename.

### 3. Two cron-output scripts re-pathed

`sartor/memory/wifi/wifi-health-monitor.py` now writes `wifi-health-{stamp}.md` to `inbox/rocinante/_memos/wifi-health/` instead of inbox root. The `.wifi-health-state.json` state cache stays at inbox root (gitignored, curator already skips dotfiles).

`sartor/memory/machines/check-registry.py` now writes `registry-drift-{stamp}.md` to `inbox/rocinante/_memos/registry-drift/`. Verified end-to-end with a manual run — new file appeared at the new location.

The scheduled tasks (`Sartor WiFi Health Monitor` every 15 min, `Sartor Registry Drift Check` every 4h) will use the new paths automatically; no Windows Task Scheduler change needed.

## Convention going forward

- **Extractor output** (curator input) → `inbox/{peer}/proposed-memories/{YYYY-MM-DD}/{ce-stamp-hash}.md` with full required-fields frontmatter.
- **Peer memo** (anything else from a peer machine) → `inbox/{peer}/_memos/{maybe-subdir}/{filename}.md`. The frontmatter is free-form; curator skip-walks the whole `_memos/` tree.
- **Cron-task output that lands in the inbox for visibility** → `inbox/{peer}/_memos/{task-name}/` subdir per cron (e.g., `_memos/wifi-health/`, `_memos/registry-drift/`).
- **Curator-private state** → underscore-prefix subdir at `inbox/{peer}/_{name}/`. The curator's `RESERVED_DIRS` should include each.

If a peer machine wants to send a memo to another peer, write to that peer's `_memos/`. Curator stays out.
