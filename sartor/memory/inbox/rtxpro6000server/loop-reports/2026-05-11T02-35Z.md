## Wiring fix this wake — vast_metrics.service

### Anomaly

`vast_metrics.service` was in a 10-second restart loop with `status=203/EXEC` (permission denied) since **2026-05-04T03:13:47Z**. Restart counter at fix time: **58,807**. The unit file is at `/etc/systemd/system/vast_metrics.service` (vast.ai-managed) and runs `/var/lib/vastai_kaalia/latest/launch_metrics_pusher.sh` as user `vastai_kaalia`. The script existed and was readable; permission was `-rw-r--r--` (644). Every other sibling `launch_*.sh` in the same directory had `-rwxr-xr-x` (755). The package shipped this one file without the execute bit.

### Diagnosis

`/var/lib/vastai_kaalia/latest -> version_299` (symlink ctime 2026-05-04, which matches the first failure). The kaalia package's `version_299` release shipped `launch_metrics_pusher.sh` without `+x`. The script is a 3-line wrapper: `#!/bin/bash` / `cd "$(dirname "$0")"` / `exec python3 machine_metrics_pusher.py`. No upstream code-level bug — just a packaging-bit miss.

### Fix applied

```bash
sudo chmod +x /var/lib/vastai_kaalia/latest/launch_metrics_pusher.sh
sudo systemctl restart vast_metrics
```

Result: service `active (running)`, `python3 machine_metrics_pusher.py` is PID 641970 and has been alive ~30s+ as of report time; no further EXEC failures in the journal since 02:35:12Z. `NRestarts=58807` is now frozen.

### Why this was in scope per §14a (proposed) wiring grant

The peer-self-loop prompt names "minor script bugs that prevent execution... when the fix is mechanical and the spirit of the script is preserved" as in-scope. A `chmod +x` on a sibling-pattern-matching shipped script is the cleanest mechanical fix possible — no logic change, no policy change, just restores an obvious-from-context bit. The script is owned by vast.ai's package not by Sartor, but the prompt's wiring grant doesn't constrain ownership; it constrains intent-preservation. Intent is fully preserved.

### What will happen on next vast.ai package update

When vast.ai ships `version_300` and the `latest` symlink moves, my `chmod` evaporates because it lives on the old version_299 inode. If they ship the same packaging bit, the bug returns. The structural fix is upstream — Alton or the household may want to file a packaging bug with vast.ai support or add a Sartor-side cron that re-applies `chmod +x` after `version_*` changes. **Surfacing the choice; not adding a cron autonomously.**
