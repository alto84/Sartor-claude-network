The 2026-05-04 truth audit (committed `d5e7e1ac`) reported gpuserver1's on-demand rate as $0.30 and flagged it as drift from documented $0.35-$0.40. Live state today is $0.35 / $0.30 (min_bid 0.30, listed 0.35), end_date 2026-09-10. Reconciling the two:

- `2026-05-05T2000Z`: listed=false, rented=true (machine unlisted while rental continues)
- `2026-05-05T2030Z`: listed=true, rented=true (re-listed 30 min later)

The 30-min unlist-relist window on 2026-05-05 is when the price was changed via the web UI or CLI — not by `vastai-tend.sh` (which only reads, doesn't write). Hypothesis: someone (Alton or Rocinante via UI) raised the listed rate from $0.30 back to $0.35, raised min_bid from $0.25 to $0.30, and shortened end_date from 2026-10-24 to 2026-09-10, all in one edit. The ratio $0.30→$0.35 is +17%, min_bid $0.25→$0.30 is +20% — narrowing the spread between listed and floor.

So the 2026-05-04 audit caught a temporary low-price window, not a permanent move. **My audit's "STALE — $0.35 documented, $0.30 live" framing was wrong-direction**: docs were correct at the time of writing; live had drifted lower temporarily and corrected itself a day later. The right framing is: pricing changes via UI bypass the `vastai-tend.sh` write path, so docs and live can diverge in either direction without anyone noticing.

**Skill-relevant takeaway for `vastai-management`:** the per-30-min state-change file only records `listed` and `rented` booleans, not the price. A price change via UI looks identical to a price-unchanged relist. If we want price drift detection, `vastai-tend.sh` should also write the four pricing fields (`listed_gpu_cost`, `min_bid_price`, `listed_storage_cost`, `end_date`) into the state cache, and emit a separate event when those change. Proposing as a follow-up; not fixing it autonomously this wake (script edit is mechanical but it changes the inbox-event vocabulary, which is downstream-consumed by the curator).
