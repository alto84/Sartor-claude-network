# Architecture

How the pieces of the safety research wiki fit together. Read this once before customizing anything.

## Mental model

```
                          ┌─────────────────┐
                          │  Your sources   │
                          │  (papers, docs, │
                          │   emails, db    │
                          │   queries)      │

                                   │

                                   │ (write-heavy,
                                   │  updates 10-15 pages)
                                   ▼
          ┌────────────────────────────────────────────┐
          │                                            │
          │         Markdown files on disk             │
          │  (the wiki -- the durable artifact)        │
          │                                            │
          │   ┌─────────┐  ┌─────────┐  ┌─────────┐   │
          │   │mechanism│  │  drug   │  │   AE    │   │

## The three layers

**Layer 1: The substrate** — markdown files on disk. This is the durable artifact. Everything else is derived or transient. If you lose `wiki.py` and the `indexes/` directory, you can regenerate them. If you lose the markdown files, you lose the wiki.

**Layer 2: The query layer** — `wiki.py`. Reads the markdown files, builds indexes, answers queries. Pure read-side machinery. Never writes to the markdown files except through the user's edits. Stateless between runs (except for the derived indexes, which are regenerable).

**Layer 3: The agent layer** — the `wiki-reader` subagent and the `wiki-reindex` scheduled task. These are orchestration patterns an LLM agent uses to keep the system clean and answerable. They wrap `wiki.py` with behavioral rules ("never read raw files, always use the query layer", "run lint weekly", "preserve frontmatter on ingest").

## Why this split matters

The substrate is durable (markdown), the query layer is fast (Python + indexes), and the agent layer is intelligent (LLM judgment). Each layer has one job. You can replace any layer without breaking the others:

- Swap the query layer for something faster (e.g., rewrite in Rust) — the markdown files don't change
- Swap the agent layer for a different LLM provider — the query layer and markdown don't change
- Swap the markdown files (e.g., migrate to a new vault structure) — re-run the query layer, regenerate indexes

## The two spine files

Every wiki has two durable spine files that every operation touches:

- **`index.md`** — categorized catalog. Every entry is a wikilink + one-line summary. Updated on every ingest. The first thing a reader (human or agent) looks at.
- **`log.md`** — append-only chronological ledger. Every entry has a `## [YYYY-MM-DD] action | title` prefix parseable with grep. Never rewrite, never reorder.

These files are NOT generated by `wiki.py --reindex`. They're living curated files that the human and the agent maintain together. Losing them is a real loss.

## Generated indexes

Everything in `indexes/` is derived. Regenerable via `wiki.py --reindex`. Safe to delete. Should be gitignored (except `_index.md`, which is the Obsidian-visible entry point to the directory).

| File | Purpose |
|------|---------|
| `backlinks.json` | Reverse wikilink index: `{target: [sources]}` |
| `tag-index.json` | Tag to files map |
| `orphans.json` | Files with zero incoming wikilinks (excluding meta/reference types) |
| `broken-links.json` | Wikilinks whose targets don't exist |
| `similarity.json` | Top-5 semantically similar files per page (if embeddings available) |
| `_index.md` | Obsidian-visible entry point |

## The bounded health file

`state/wiki-state.md` is a Hermes-pattern bounded memory file capped at 1500 chars. It contains the current health snapshot: file count, tag count, orphan count, broken link count, tier distribution, top 5 orphans, top 5 broken links. Overwritten on every reindex.

Agents can read this file on context injection without blowing up their token budget. A full `wiki.py --health` dump can be much larger; `wiki-state.md` is the compressed version.

## Reindex algorithm

```
1. Collect .md files (skip SKIP_DIRS: .obsidian, .git, __pycache__, node_modules, .meta, .index, indexes)
2. Parse each file:
     a. Extract YAML frontmatter
     b. Strip fenced code blocks from body (so documentation placeholders don't count as wikilinks)
     c. Extract wikilinks, callouts, headings, title
3. Build backlinks:
     For each file, for each wikilink target, resolve to canonical rel_path,
     then add (source, target) to backlinks[target]
4. Build tag-index: for each file, for each tag in frontmatter.tags, add file to tag-index[tag]
5. Find orphans: files (excluding meta/reference/feedback types) with no entries in backlinks
6. Find broken links: wikilinks that don't resolve to any file (stem, rel_path, or basename match)
7. (Optional) Build similarity: for each file, query embeddings for top-5 similar
8. Write all indexes to indexes/
9. Write _index.md (Obsidian entry point) with summary stats
10. Write state/wiki-state.md (bounded health file)
```

Wikilink resolution tries in order: exact stem match → rel_path suffix match → basename extraction from path-style targets. This handles both `[[CRS]]` and path-style `[[subdir/file]]` wikilinks.

Code blocks are stripped before wikilink extraction so documentation placeholders like `[[FILE]]` inside ``` fences don't pollute the broken-link count.

## Provenance via git

If the wiki directory is a git repo, `wiki.py --article <file>` includes:
- Last commit date and author
- Commit message of the last change
- Count of commits touching the file in the last 7 days

This is optional. If git isn't available, provenance falls back to the frontmatter `updated` and `updated_by` fields.

## Semantic similarity (optional)

The wiki layer calls a `SemanticSearch` class from `embeddings.py` if the module exists. The integration is loose — if the module isn't available, similarity is skipped with a warning.

To add semantic similarity:
1. Write an `embeddings.py` module next to `wiki.py`
2. Expose a `SemanticSearch` class with a `search_semantic(query, top_k)` method
3. Implement it using your preferred embedding provider (sentence-transformers offline, a corporate API, OpenAI, etc.)
4. Run `wiki.py --reindex` — it will pick up the module and populate `similarity.json`

The wiki is deliberately agnostic about which embedding model you use.

## Data flow: single page update

A concrete walkthrough of what happens when you edit `tisagenlecleucel.md` and run reindex:

```
1. You edit tisagenlecleucel.md to add a new adverse event entry
   - Add [[QT-prolongation]] to the Known Adverse Events section
   - Bump frontmatter `updated: 2026-04-09`
   - Save
2. You run `python wiki.py --reindex`
3. wiki.py collects all .md files, re-parses tisagenlecleucel.md
4. It extracts the new wikilink to QT-prolongation
5. It rebuilds backlinks: now QT-prolongation.md has tisagenlecleucel.md in its wikilinks_in
6. It writes updated backlinks.json, tag-index.json
7. It writes state/wiki-state.md with the new counts
8. Next time you run `wiki.py --article QT-prolongation`, the backlinks panel shows tisagenlecleucel
9. Next time you run `wiki.py --article tisagenlecleucel`, the article view shows the QT-prolongation wikilink
```

## Data flow: ingest a paper

```
1. You drop a PDF in sources/ or paste the abstract into the agent
2. The agent runs through the ingest workflow (see docs/INGEST.md):
   a. Extract entities: drugs, mechanisms, AEs
   b. Find existing pages for each entity
   c. Update each page in place with new claims
   d. Create new pages for entities without existing pages
   e. Append a log entry to log.md
3. You run `wiki.py --reindex`
4. Backlinks and tag index refresh
5. You run `wiki.py --lint` to confirm no new orphans or broken links
6. You commit to git if using
```

## Data flow: answer a question

```
1. You or another agent asks: "What are the known safety signals for CAR-T?"
2. The wiki-reader subagent (or you) runs:
   a. `wiki.py --tag drug/car-t` -> lists all CAR-T drug pages
   b. For each drug page: `wiki.py --article <drug>` -> gets structured view with AEs and signals
   c. `wiki.py --tag signal/active` -> lists active signals
   d. Intersect: signals that mention CAR-T drugs
3. The reader returns a focused answer with file:line citations
4. The parent agent never loads raw markdown
```

## Extension points

Where to add features cleanly without touching the core:

- **New query type** → add a method to the `Wiki` class and a CLI flag in `main()`
- **New index** → add a build function, call it from `reindex()`, write to `indexes/`
- **New article_view field** → add to the dict returned by `Wiki.article_view()`
- **New page type** → add a template to `templates/`, add the type to the frontmatter vocabulary in `conventions/CONVENTIONS.md`
- **Semantic similarity** → implement `embeddings.py` with a `SemanticSearch` class
- **Ingest from a new source type** → add a prompt template to `docs/INGEST.md`, no code changes needed
- **Web UI** → write a FastAPI app that imports `wiki.Wiki` and exposes article views over HTTP

## Obsidian compatibility (why it matters)

The wiki uses Obsidian-native syntax throughout:
- YAML frontmatter → rendered in Obsidian's Properties sidebar
- `[[wikilinks]]` → clickable, populate graph view
- `> [!callout]` → rendered with icons in reading view
- Hierarchical `#tags` → appear in the Tags pane

This means the same folder works as:
- A Python-queryable knowledge base via `wiki.py`
- A visual Obsidian vault for human browsing
- A git-tracked markdown corpus for any other tooling

You don't need Obsidian. But if you have it, everything works without configuration.

## What this architecture does NOT solve

- **Multi-user concurrent editing** — single-author or small-team with soft coordination only. No real-time sync, no conflict resolution UI.
- **Real-time signal detection** — the wiki is a navigation and synthesis layer, not a monitoring system.
- **Schema evolution** — if you want to add a new required frontmatter field after the wiki is populated, you'll have to migrate existing files manually.
- **Rollback** — the wiki relies on git for version history. If you're not using git, there's no built-in rollback.
- **Access control** — the wiki is a folder of files. Permissions are filesystem permissions.

These are deliberate non-goals. The wiki is a research tool, not a document management system.
