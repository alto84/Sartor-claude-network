- **2026-04-11 T+13:20 [gpu-operator]** **sft-v1 template-leak bug diagnosed and sft-v2 retrain live with the fix.** Evidence: 83% leak rate (15/18 rows) on final-adapter C1 eval, leak also present at checkpoint-100 in mid-response positions. Root cause is NOT a tokenization mismatch (text and chat-template forms tokenize to identical IDs, verified). It's **loss-masking**: SFTTrainer with `dataset_text_field="text"` was computing causal-LM loss over the entire rendered chat transcript including the `<extra_id_1>User\n` and `<extra_id_1>Assistant\n` role-delimiter tokens. The model learned to emit those as legitimate continuation targets; at inference time it generates an assistant reply, emits `<extra_id_1>User\n` as a turn separator, hallucinates a user turn, and enters a multi-turn bleed-through loop. Position-varying leak (start-of-response vs mid-response) reflects how long the assistant reply runs before the model decides "time for the next turn." Matches every observed symptom including the degenerate-repetition rows where the model alternates fake-user / fake-assistant until max_new_tokens.
  - **Fix implemented in `scripts/train_sft.py`:** (a) new `build_messages_dataset()` passes raw conversational `messages` rows to SFTTrainer, (b) new `--assistant-only-loss` CLI flag sets `SFTConfig.assistant_only_loss=True`, (c) new `patch_chat_template_for_assistant_masking()` monkey-patches the tokenizer's chat template to add `{% generation %}...{% endgeneration %}` wrappers around the assistant content branch (Nemotron-Mini-Instruct ships without them and TRL's assistant-only-loss silently no-ops if they are absent), (d) `--save-strategy epoch --save-total-limit 12` so every epoch is retained naturally.
  - **Monkey-patch details for reproducibility:** patches `tokenizer.chat_template` in memory only. Before: `{% elif message['role'] == 'assistant' %}{{ '<extra_id_1>Assistant\n' + message['content'].strip() + '\n' }}`. After: `{% elif message['role'] == 'assistant' %}{{ '<extra_id_1>Assistant\n' }}{% generation %}{{ message['content'].strip() + '\n' }}{% endgeneration %}`. The pre-match check raises RuntimeError if the template changes upstream, so this won't silently no-op on a tokenizer update.
  - **Mask verification on one real corpus row (constitution chunk 0):** 436 total tokens, 377 with mask=1 (assistant content starts at token 59 which is `'---'`, the constitution frontmatter opener), 59 with mask=0. `<extra_id_1>` token ID 5 appears at both User and Assistant markers and **both have mask=0**. Confirmed only reply content tokens contribute to loss.
  - **Corpus-wide mask coverage (all 136 training rows):** 44,052 assistant-content tokens (mask=1) / 50,589 total tokens = **87.1% coverage of loss computation.** 6,537 tokens (12.9%) masked out, corresponding to system prompts, user prompts, and `<extra_id_1>User` / `<extra_id_1>Assistant` / `<extra_id_0>System` role delimiters. Coverage is below the 90% floor lab-lead set but deliberately so — the constitution recitation corpus has genuine system and user wrappers around each constitutional passage, and those should NOT contribute to loss. A corpus with more system/user content would show lower coverage and that would be correct behavior, not a bug.
